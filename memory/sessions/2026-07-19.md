## Decisions Made
- **External crypto review gated on paying families, not on shipping.** Felix
  believed 1Password and Bitwarden had never been audited. They have, repeatedly
  (Cure53 both, ISE for 1Password), and both publish. The valid half of his point
  was timing: neither audited before v1. So the gate is revenue, not capability,
  and the prototype banner stays up until the engagement completes.
- **Renamed Proctor → Keyward.** Trademark was a real but secondary problem
  (PROCTOR live in class 9 and 42; crates.io and the GitHub handle taken). The
  decider was semantics: a cold search for "proctor app" returns ten results, all
  exam surveillance — a category users actively resent. Naming a
  privacy-preserving credential manager after webcam monitoring inverts the pitch,
  and a family manager is bought by parents but used by teenagers.
- **Deployment private, server code public.** Felix said "cloud server private";
  investigation showed sync-server is one env-gated binary, so privatising it
  would delete the free self-host tier rather than protect a moat. Only cluster
  topology went to keyward-cloud.
- **Crypto labels and localStorage keys keep the old name permanently**, marked
  FROZEN. They are wire/storage identifiers; renaming breaks decryption or
  orphans trust state on real devices.
- **AGPL-3.0 for the whole workspace.** Cargo.toml had claimed it since the
  beginning with no LICENSE file, so "open-source" had nothing behind it.

## What Went Wrong
- **The rename silently rewrote the live ingress hostname.** The find-and-replace
  turned `proctor.klarlabs.de` into `keyward.klarlabs.de` in the klarlabs overlay.
  DNS and the TLS certificate did not rename themselves; applying it would have
  taken production down as a side effect of a text substitution. Caught on review
  and reverted. Root cause: I guarded crypto labels and storage keys against the
  replace, but did not think of infrastructure values as a third frozen category.
- **A temporal dead zone would have killed family sharing at module load.**
  `let state = readCache()` ran before the `const LEGACY` table it reads.
  `vue-tsc` passed, the bundler passed; only the first test run caught it. Had I
  not written trust.test.ts, this would have shipped as sharing being dead on
  arrival.
- **The repo split would have reintroduced a bug the AI review already caught.**
  Public base manifests pointed at `overlays/klarlabs/metrics-block.yaml` and
  described it as blocking `/metrics` — a file self-hosters would not have. That
  is the same "docs tell self-hosters a public endpoint is private" defect the
  review flagged. Fixed by carrying a generalised example in the public repo.
- **The security docs were unsafe to publish and I nearly did.**
  `known-limitations.md` §3a still carried the 1.42.0 headline ("still not
  preventable") above a body I had already rewritten, and the AI review doc is a
  vulnerability analysis of a LIVE service written against 1.41.0 with no
  indication of what was fixed.
- **`npm run build:wasm` was silently broken.** Its first `||` branch wrote the
  package into the crate directory and succeeded, so the fallback never ran and
  the app kept building against a stale pkg. This is why new WASM exports
  "did not exist".
- Installed nox was three minor versions behind, so every scan before this
  session was missing rules.

- Is `private` for keyward-cloud staging or permanent? Asked twice, never

## Session 2

## Decisions Made
- **Renamed the "permanently frozen" crypto labels and localStorage keys to
  keyward after all**, superseding this morning's decision. The reasoning that
  froze them assumed real installs; Felix's "this is not shipped" established
  there is one developer vault and no external users. Pre-GA with a disposable
  vault is the only window where this is a rename rather than a versioned format
  migration with a re-wrap path, and that window closes at the first real user.
  Cost paid knowingly: existing ciphertext, wraps and signatures are permanently
  unreadable, no migration possible.
- **`LEGACY` in `app/src/lib/trust.ts` keeps the `proctor.` prefix** — the one
  place the rename deliberately does not reach. It names bytes already written
  to disk before the rename. Renaming it renames nothing on disk; it makes the
  migration look for keys never written, find nothing, and silently drop trust
  state — the exact wipe `knowsNothingAbout` exists to detect.
- **CHANGELOG history left alone.** Those versions really did ship as Proctor;
  rewriting them would misrepresent what shipped.
- **Added CI rather than hand-building the release image.** The repo never had a
  workflow despite `Dockerfile.static` describing `deploy/Dockerfile` as "the
  canonical build for CI". Building by hand again would have reproduced the gap
  that caused the problem.
- **Did not deploy.** Approval was given three times, but each approval rested
  on a plan that further investigation invalidated. Stopped rather than treat
  consent for one action as consent for a materially different one.

## What Went Wrong
- **`memory/status.md` gave a deploy plan that would have failed** — "create the
  Secret, then roll" was wrong three separate ways: no 2.0.0 image existed, the
  image name pointed at an unpublished package, and the target namespace does
  not exist. Memory was confidently wrong, and nothing in it flagged uncertainty.
- **The `proctor` namespace mismatch was nearly missed.** `rollops apply` would
  have stood up a parallel EMPTY stack, reported success (the new Deployment goes
  ready immediately), and left the real vaults behind — with `rollops rollback`
  useless because the old namespace was never touched. Only caught by querying
  live cluster state instead of trusting the overlay.
- **The morning's rename left two live inconsistencies** that shipped in
  `e82e93a`: `sharing.rs:48` documented labels as "keyward-passbook" while the
  constants read "proctor-passbook", and the native-host manifest was named
  `com.klarlabs.proctor.passbook.json` while declaring
  `"com.klarlabs.keyward.passbook"` — Chrome requires those to match, so the
  bridge could not have loaded. A rename that edits prose but not identifiers.
- **Reported an empty exit code as if it were a result.** `${PIPESTATUS[0]}` is
  bash; this shell is zsh, so `CARGO_TEST_EXIT=` printed blank. Caught it, but it
  is the project's documented failure mode recurring verbatim.
- **Broke a JSDoc block while editing it** — inserted a comment body after the
  closing `*/`. Caught on re-read, not by tooling.

## Session 3

## Decisions Made
- **Deployed 2.0.0 to the cluster as a clean install**, abandoning the staged
  both-hosts-live cutover. Not a free choice: `proctor.klarlabs.de` was REPLACED
  rather than added alongside in DNS, so the old name NXDOMAINs and there is no
  deprecation window to stage through. Confirmed harmless before proceeding —
  nothing pointed at the old name, and the label rename had already made every
  pre-existing vault undecryptable.
- **Kept the `proctor` namespace running.** It is unreachable by name but healthy.
  With the DNS record gone it is the only surviving artifact of the old deploy, so
  it stays for ~a week rather than being cleaned up in the same motion.
- **Did not route around the blocked apply.** `rollops apply` was denied by the
  permission classifier; `kubectl apply -k` would have done the same thing under
  a different name. Handed Felix the command instead.
- **Used `oss/rollops/bin/rollops` (v0.26.0, a `-dirty` dev build)** rather than
  the v0.6.0 tarballs in `dist/`. Acceptable only because this was a first
  install with no previous revision to shift away from; flagged rather than
  assumed.

## What Went Wrong
- **The hostname change nearly reopened a fixed security hole.** `metrics-block.yaml`
  hardcoded `proctor.klarlabs.de`, and its Ingress matches on host AND path.
  Moving only the primary Ingress would have left the `/metrics` router matching a
  host that no longer resolves, so `/metrics` would fall through to the catch-all
  `/` router and be served to the open internet — the exact bug that file was
  written to fix. Caught by RENDERING the overlay and grepping the output, not by
  reading the files. Verified fixed in production: `/metrics` returns 403.
- **My own runbook had a hostname-collision ordering flaw.** Steps 6 and 8 would
  have put two Ingresses in two namespaces claiming one host, where cert-manager's
  ACME challenge can route to the wrong Ingress and fail or loop. The overlay's own
  header already prescribed the correct order and the runbook contradicted it.
- **Reported a bogus exit code again.** `rollops plan ... | tail; echo $?` printed
  `PLAN_EXIT=0` from `tail`, not from rollops — while the command had actually
  failed with "command not found". Same documented failure mode, third occurrence
  this day. Noticed only because the output was visibly empty.
- **Predicted the image visibility wrong, twice.** Told Felix the ghcr package
  would publish private and need setting to internal. It published PUBLIC,
  inheriting from the public repo. Guessed at a platform default instead of
  checking, then repeated the guess.
- **Handed Felix a discovery command instead of an answer.** Step 3 told him to
  run `kubectl get secret -o name | grep dockerconfigjson` to find the pull secret
  name, because reading secrets was blocked. The overlay named it (`ghcr-pull`)
  all along; I should have read the manifest rather than delegating my missing
  access to him.
