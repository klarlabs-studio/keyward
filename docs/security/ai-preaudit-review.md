# PRE-AUDIT HARDENING REPORT — Proctor
**Scope:** `crates/passbook`, `crates/crypto`, `crates/sync-server`, `crates/sync`, `crates/sync-postgres`, `app/src/lib`, `deploy/`
**Method:** source read against HEAD; every claim below was re-verified in the file, not taken from docs or from the candidate write-ups.
**Purpose:** ensure a paid auditor's hours go to novel findings. This is not a substitute for that audit.

---

## 1. Bottom line

**Do not pull the deployment. Do warn the single current user, and turn off — or stop relying on — family sharing until findings F1–F3 are fixed.**

The split is clean and worth stating plainly:

- **The personal vault is sound.** `sealing.rs` derives `K = SHA-256(argon2id(master) ‖ secret_key)` over fixed-length operands, fresh 16-byte salt and fresh 24-byte nonce per seal, XChaCha20-Poly1305, all randomness from `OsRng`. I enumerated every AEAD seal site and every DH site: no nonce reuse path, no counter- or content-derived nonce, no unguarded DH. The relay genuinely holds only ciphertext + salt for personal vaults. If a user never touches family sharing, server compromise reveals nothing.
- **Family sharing does not hold against its own stated threat model.** The client automatically wraps the shared vault key `K_vault` to any public key the relay reports, with no verification and no user gate. A malicious or compromised relay reads every shared credential. This is a real break of "compromising the server must not reveal any vault content" — for shared content only.

Why this does not warrant pulling the service: the attacker required is *the relay operator*, which is the user themselves; there is one user; the personal vault is unaffected; and Proctor's own `ADR-0004:137` already names key substitution as a **"Known open item (must be closed before GA)."** The honest framing is that the pre-GA gate was written down and then the managed instance shipped anyway. That is a release-governance failure, not a surprise vulnerability — and it is the single most important thing to fix before a second user exists.

One non-crypto item deserves separate mention: **the passphrase generator produces 22–59 bits of entropy while the UI reports 82–275 bits and the security dashboard rates every one "Excellent"** (F9). No relay involvement, no key material — but a password manager affirmatively misreporting the strength of credentials it generated is a user-facing correctness failure, and I measured it rather than inferred it.

## 2. Confirmed findings, ranked by real-world severity

### F1 — `loadFamily` auto-wraps `K_vault` to any relay-supplied public key `[critical]`

**Defect.** On every open of a family vault, the client grants access to every directory entry the *relay* reports as lacking a wrap, using the *relay-supplied* public key, with no confirmation and no verification. `grant_access` authorizes only the granter and validates the recipient not at all.

`app/src/lib/sharing.ts:517-534` (verified verbatim):
```ts
const wrappedIds = wrappedMemberIds(keys.json);
const missing = group.members.filter((m) => !wrappedIds.has(m.member_id));
for (const m of missing) {
  updated = grant_group_access(updated, member.secret, member.id,
    JSON.stringify({ id: m.member_id, name: m.name, public_key: m.public_key }));
}
const newVersion = await putKeys(groupId, updated, keys.version);
```
`group` is the raw `GET /v1/groups/{id}` response. `crates/passbook/src/sharing.rs:394-400` — `grant_access` calls `unwrap_for(existing_member)` then `wrap_to(&vault_key, new_member)`; the only recipient check anywhere is `checked_secret` rejecting low-order points, which an attacker with an honest keypair sails past. The code comment at `sharing.rs:326-327` concedes the provenance: *"The recipient key came from the relay."*

**Attack.** The relay appends one fabricated member row with an X25519 key it holds. The next time any member with access opens the vault, their client wraps `K_vault` to that key and PUTs it back. The relay unwraps and calls `open_content` on the blob it already stores. Full plaintext of every shared credential. No user interaction beyond normal use.

**Contradicted claim.** `ADR-0004:130` — *"it grants no access until an existing member wraps to it, **which is a human decision**."* It is an unattended background action. `crates/sync-server/src/main.rs:75-76` claims the server "sees only public keys" while the code lets the server *choose* those public keys.

**Fix.** Do not derive the grant set from the relay's directory. Pin member public keys trust-on-first-use in `GROUPS_STORAGE` (currently only `{groupId, name}` is persisted — verified) and refuse to wrap to a key that differs from the pinned one without an explicit, blocking user confirmation showing the old and new fingerprints. Longer term: member-signed directory entries, which is the mitigation ADR-0004 already names.

### F2 — Wrapped-key and content blobs carry no sender authentication, and the safety number cannot cover them `[high]`

**Defect.** `wrap_to` (`sharing.rs:320-353`) needs only the recipient's public key; the AEAD call passes plaintext with **no associated data** and no signature. `unwrap_for` (`sharing.rs:358-386`) accepts any wrap under a matching `member_id` that decrypts to 32 bytes. The client never pins a vault key across loads.

Crucially, `safety_number` (`sharing.rs:104-117`, verified) hashes `SAFETY_NUMBER_INFO`, then per sorted member `len(id) ‖ id ‖ public_key` — **and nothing else.** `wrapped_keys` and `content` are structurally outside the digest.

**Attack.** The relay generates `K_evil`, wraps it to the real member public keys, and overwrites both blobs. Every member unwraps and decrypts successfully. **The safety number is unchanged**, so a family diligently comparing fingerprints out of band detects nothing. The first subsequent `saveFamilyEntries` re-seals all shared credentials under `K_evil` (`sharing.ts:570-571` re-unwraps from the relay-served blob with no comparison to any prior key). At group creation this is completely invisible: `createFamilyVault` seals `JSON.stringify([])`, so the owner decrypts an empty list exactly as expected.

**Contradicted claims.** `ADR-0004:131` — *"a tampered wrap fails `unwrap_for`."* A wholly substituted wrap does not fail; it succeeds. `docs/security/threat-model-passbook.md:111` — relay *"Cannot: forge a wrap that a member's key opens."* It can.

**Fix.** Bind wraps: pass `group_id ‖ member_id ‖ epoch` as AEAD associated data in `wrap_to`/`unwrap_for`, and extend `safety_number` (or add a separate group-state digest the client pins) to cover the wrapped-key set. Store a monotonic version floor client-side so rollback is detectable.

### F3 — The designated mitigation runs after the key has already left `[high]`

**Defect.** The safety number is (a) computed from the same relay-served directory, (b) never read by any branch, and (c) produced *after* the grant in F1 has already uploaded the wrap.

Verified: `sharing.ts:499` sets `safety: await safetyNumber(group.members)` onto `base`; the grant loop and `putKeys` at `:517-534` execute before `loadFamily` returns; `ShareDialog.vue:262` renders it as advisory prose. `grep` across `app/src` finds no consumer other than that template. There is no trust-on-first-use pinning anywhere. `justGranted` (`share.ts:172-178`) is a toast fired after the upload — and `name` is attacker-supplied at join (`main.rs:1159`, verified), so an injected member can be labelled "Mom."

**Consequence.** A user who follows the in-app instruction exactly — compare the number, "if it differs, stop and don't share anything" — stops *after* `K_vault` is already in the relay's hands. The control of record cannot prevent the attack it exists for.

**Fix.** Block on comparison before the first grant to any newly-seen key. Persist "this number was confirmed at version N" and alert on change. This is the same fix as F1; I list it separately because an auditor should be told the mitigation is structurally, not incidentally, too late.

### F4 — Revocation is undone end-to-end by any outstanding invite; invite TTL is attacker-chosen and unbounded `[high]`

**Defect (three verified links).**
1. `crates/sync-server/src/main.rs:1109-1119` — `.get("ttl_seconds").and_then(|v| v.as_u64()).filter(|&t| t > 0).unwrap_or(24*60*60)` then `expires_epoch: now.saturating_add(ttl)`. **No upper bound.** `ttl_seconds: u64::MAX` saturates to `u64::MAX`, so `apply_redeem`'s `now_epoch >= invite.expires_epoch` can never fire — a permanent readmission ticket. The doc comment at `main.rs:1179` still speaks of "our 24h TTL."
2. Removing a member never invalidates invites. `crates/sync/src/groups.rs:346` is `g.members.retain(...)`; the Postgres adapter is `DELETE FROM group_members`. `group_invites` is untouched in both.
3. `apply_redeem` (`groups.rs:262-283`, verified) gates only on code-hash, `redeemed_by`, and expiry. **No removed-account check.** Group removal does not revoke the account's device token, so the ex-member can still call the endpoint.

**Attack.** An Admin mints an invite for themselves with an enormous TTL before removal. They are removed; the family rotates. They re-POST the stashed code, land back in the directory, and F1's auto-reconcile grants them the *post-rotation* key on the next member load. They regain plaintext access to everything, including material added after their eviction.

**Contradicted claim.** `sharing.ts:596-597` and `ADR-0004:120-123` — *"The removed member keeps only what they had already read."* Not true on this path.

**Fix.** Clamp `ttl_seconds` (24h max). Invalidate all outstanding invites for a group on any member removal. Maintain a per-group removed-account list and reject redemption from it. Independently, fix F1 so re-entry does not silently yield the key.

### F5 — Recovery contact seals a 2SKD factor to a relay-supplied key, reaching the *personal* vault `[high, dependent on F1]`

**Defect.** `app/src/lib/sharing.ts:365-369` seals the device Secret Key to `contact.public_key`, taken verbatim from the relay-served directory, with no verification and no gate (`ShareDialog.vue:307` is a bare click handler).

**Attack.** Requires F1 first — the SealedBox lives inside the `K_vault`-encrypted ContentBlob, so the relay needs `K_vault` before it can extract it. Given F1, that is free. The relay then holds the victim's 128-bit device Secret Key **and** their personal `SealedVault` (same host serves `/v1/vault`). 2SKD collapses to a single offline guess of the master password against `Argon2::default()` — `crates/crypto/src/lib.rs:60`, m=19456 KiB, t=2, p=1, no pepper.

**Contradicted claim.** `ADR-0004:83-85` and `:133` — server *"never sees `K_vault`, any master password, or Secret Key."* `known-limitations.md` §10a discloses the shape of this but attributes the residual to §1 (low-order points), which the code has since fixed — so as written it reads as closed.

**Fix.** Fold into F1's key pinning, and additionally hard-gate the recovery flow on an explicit safety-number confirmation. This is the one path that escalates a shared-vault break into a personal-vault break, so it warrants the stricter gate even after F1 is fixed.

### F6 — Invite redemption overwrites the Owner's row, permanently orphaning the group `[medium]`

**Defect.** `apply_redeem` (`groups.rs:273-281`, verified) does `*existing = new_member` on an `account_id` match — replacing role, `member_id`, and `public_key` wholesale — while `handle_group_join` hardcodes `role: Role::Member` (`main.rs:1162`, verified).

**Attack.** An Admin mints an invite and socially engineers the Owner into redeeming it ("rejoin from your phone" — the case `apply_redeem`'s own comment blesses). The Owner's row becomes `role: Member`. `can_change_roles` is strictly `*self == Role::Owner`, so `handle_group_set_role` now 403s for **everyone, permanently** — no Owner can ever be restored. The `"an owner cannot be removed"` guard (`main.rs:1215-1218`) now matches nobody, so the Admin removes the former Owner. There is no `DELETE /v1/groups/{id}` route (verified in the dispatch table). The state is unrecoverable.

**Contradicted claim.** `main.rs:1259-1262` documents *"An Owner's role cannot be changed (no demoting/capturing the owner)"*; the join path changes it with no role check. `threat-model-passbook.md:194` rates this row **Mitigated**.

**Fix.** In `apply_redeem`, preserve the existing row's `role` on re-join (and arguably reject redemption by an existing member outright). Add an invariant check that a group always has ≥1 Owner.

### F7 — `member_id` uniqueness is unenforced by the policy layer `[medium — self-host only]`

**Defect.** `member_id` is client-supplied and unvalidated (`main.rs:1146-1163`); `apply_redeem` dedups on `account_id` only. `wrap_to` replaces by `member_id` and `unwrap_for` selects by it, so `member_id` is the sole cryptographic identity key.

**Scope correction the candidate lenses got wrong in both directions — verify this yourself:** `crates/sync-postgres/src/lib.rs:73` declares `PRIMARY KEY (group_id, member_id)`, and `insert_member` (`:488-503`) is a bare `INSERT` with no `ON CONFLICT`. A colliding id raises a PK violation, the transaction rolls back, and the caller gets a 500. **The managed instance runs Postgres** (`main.rs:417-433`, `PROCTOR_SYNC_PG` takes precedence — verified) and is therefore **not exploitable**. The file store (the self-host default) and memory store share the unconstrained `apply_redeem` and **are**.

**Attack (self-host).** A member re-joining with the Owner's `member_id` becomes unremovable (the Owner-guard's `.any()` matches the Owner's surviving row) and, at the next rotation, their later-appended wrap overwrites the Owner's — locking the Owner out with no removal path.

**Fix.** Enforce uniqueness in `apply_redeem`, not incidentally in one storage adapter. Change the Owner-removal guard to inspect the *target's own* row rather than `.any()` across the directory. The fact that one adapter enforces an invariant the shared policy function does not is itself the finding.

### F8 — `PUT /v1/groups/{id}/keys` gates on membership, not role `[medium]`

**Defect.** `main.rs:1339-1342` — `if !group.is_member(account) { 403 }` is the *only* authorization for both `Blob::Keys` and `Blob::Content`, versus `can_manage_members` at `:1213` and `can_change_roles` at `:1267`.

**Attack.** A plain Member (a child account) PUTs a `SharedVault` wrapping a fresh key only to themselves. Every other member including the Owner fails `unwrap_vault_key`; `loadFamily` returns early (`sharing.ts:507-511`) so auto-reconcile cannot repair it; blobs are overwritten in place with no history. The family vault is destroyed for everyone.

**Honest caveat.** Member-level write to `/keys` is *required* by the F1 auto-reconcile flow, and `ADR-0004:93` specifies it as `(member)`. A naive role check breaks onboarding. Also, `known-limitations.md:456` already discloses this — but files it under "Smaller items, still true," which materially under-rates permanent destruction of the vault by the lowest-privileged role.

**Fix.** Not a role check. Constrain the write to be additive: reject a keys blob that drops `member_id`s present in the current version, or retain prior blob versions so an Owner can roll back.

### F9 — Passphrase generator: 22–59 real bits, reported as 82–275, and Watchtower can never flag it `[high — separate lane]`

**Measured, not inferred.** I parsed the wordlist and ran the arithmetic:
```
crates/passbook/src/generate.rs — WORDS: count 170, unique 170, 7.409 bits/word
  3 words →  22.2 bits    5 words (UI default) → 37.0 bits    8 words → 59.3 bits
```
The comment two lines above the array says `~200 words` / `roughly 7.6 bits/word`. Both wrong; the code disagrees with its own comment.

`crates/passbook/src/watchtower.rs:19-38` scores by character-space × character-count — `space = 26 + 32 = 58` for a hyphen-joined lowercase phrase, ~5.86 bits *per character*, with no awareness that the characters are not independently drawn. `WEAK_BELOW_BITS = 60` (`:42`) is compared against that inflated number. Sampling the *shortest reachable* output at each setting puts the floor at ~82 reported bits — so **no generated passphrase can ever be flagged Weak at any setting**, and the UI labels a 22-bit phrase "Excellent."

**Attack.** An attacker with a site password hash from a third-party breach — the exact scenario a password manager exists to survive — enumerates against the wordlist, which is public in this open-source repo, with the hardcoded `-` separator and the UI-enforced 3–8 word range. `170³ = 4.9e6`; `170⁵ = 1.4e11`. Trivial against a fast site hash. The user was told "Excellent," and `refreshSecurity` reports zero issues, so nothing prompts rotation.

**Scope limits, stated so this is not overweighted:** passphrase mode is opt-in (`AddItemDialog.vue:23`, `ref(false)`); the default 20-char generator uses correct rejection sampling and is sound; `generate_passphrase` is never on a key-derivation path (only stored item passwords and CLI `generate`). This weakens per-site credentials, not the vault.

**Fix.** Ship a real diceware list (7776 words ≈ 12.9 bits/word) and make `strength_bits` structure-aware — have the generator return its own entropy rather than re-estimating it from the rendered string. Not disclosed anywhere in `docs/security/`.

### F10 — Rate limiter keys on the proxy address `[medium, availability]`

`main.rs:601-606` — `client_ip()` returns only `request.remote_addr()`. `grep` for `forwarded` / `x-real-ip` across `crates/` returns **zero hits**. Behind the L7 ingress every request carries the controller pod IP, so all users share one bucket. `rate_limited` is called at exactly two sites — `:655` (`POST /v1/register`) and `:975` (invite mint), verified.

One host at a few req/s permanently 429s **all** new signups and **all** family-invite minting service-wide, at negligible cost. The control is inverted into a lockout primitive: it causes a denial that would not exist if it were absent.

`deploy/README.md:155-157` documents the proxy-address keying as an accuracy caveat and prescribes `externalTrafficPolicy: Local` / `use-forwarded-headers` — **neither works**, because the server has no XFF code path at all. That doc/code disagreement means no operator-side configuration can fix this.

**Fix.** Parse `X-Forwarded-For` with a trusted-proxy hop count, or move rate limiting to the ingress (which `ADR-0005:223` already names as the plan). Existing-user paths (`/v1/vault`, device management) are unaffected — this blocks growth, not operation.

### F11 — Unbounded request bodies with no rate limit on any write path `[medium, availability]`

`main.rs:686`, `:895`, `:1362` — all three do `read_to_end` into an unbounded `Vec` with no length check (verified). No write path is rate-limited. Postgres stores these as TOASTed `BYTEA` on a 5 Gi PVC.

A single free-registered account looping 16 MiB `PUT /v1/vault` requests generates dead tuples faster than a 1-CPU-limited autovacuum reclaims them; disk exhaustion PANICs Postgres and stops all sync for every user, with manual recovery required. The edge caps bodies at 16 MiB (Traefik `buffering` middleware in `deploy/overlays/klarlabs/request-limits.yaml`) but that is a ceiling per request, not a budget.

`ADR-0004:134` lists *"cap blob and `SharedVault` size"* as the implemented DoS mitigation. It is not implemented. (`known-limitations.md:288` correctly records this; the ADR does not.)

**Fix.** Application-layer size cap on every blob write, plus a per-account write rate limit and a storage quota.

### F12 — `shuffle()` spins forever for length ≥ 257 `[low, correctness]`

`crates/passbook/src/generate.rs:73-87` (verified):
```rust
let n = i + 1;
let limit = 256 - (256 % n);      // n > 256  =>  256 % n == 256  =>  limit == 0
if (b[0] as usize) < limit { ... } // never true
```
`generate_password` clamps only the lower bound; the WASM export takes an unbounded `u32`; `passbook-cli` parses an unbounded `usize` from argv, so `proctor generate 300` hangs. Not reachable from the shipped UI (slider caps at 40) and the "attacker" is the user themselves — **not a security finding**, but a one-line bug worth fixing before an auditor spends time on it. Sibling `pick` is protected only by a `debug_assert!` that is compiled out of release.

**Fix.** `let limit = if n > 256 { 256 } else { 256 - (256 % n) };` — or draw a `u16` as `generate_passphrase` already does. Add a `length = 300` regression test; the existing test uses 200, just under the cliff.

### F13 — Documentation contradicting code, in both directions `[low, but fix before the audit ships]`

Four verified instances:

1. **`known-limitations.md` §1 and §2a describe defects the code does not have.** §1 states `was_contributory()` is *"never called."* `sharing.rs:433-438` defines `checked_secret`, invoked at all four DH sites (`:147, :170, :330, :369`) with a regression test at `:466-513`. §2a states the recovery box reuses `HKDF_INFO`; `sharing.rs:43` defines `SEALED_BOX_INFO`, used at `:147/:170`, with `assert_ne!(HKDF_INFO, SEALED_BOX_INFO)` at `:525`. `git log` shows the fix and the review package landed in the **same commit** (`f970478`) — the package was stale on publication. `review-scope.md:46` designates this the *"single item we most want a reviewer to confirm or refute"* and asks for a PoC. **You would be paying for a PoC against a control that provably fails closed.** Fix this first; it is the highest-leverage edit in this report.
2. **`deploy/README.md:206-209`** — *"keep `/metrics` **cluster-internal** (it is not routed by the ingress)."* `deploy/k8s/ingress.yaml:26-28` routes `path: /` with `pathType: Prefix`, which matches `/metrics`. `deployment.yaml:37-45` documents that the identical claim was already found false and served HTTP 200 publicly. The fix was applied to the manifest and not to the README; the README never mentions the `overlays/klarlabs/metrics-block.yaml` that actually blocks it. Disclosed data is aggregate counters + exact version/backend — low impact, but the doc affirmatively tells self-hosters a public endpoint is private.
3. **`ADR-0004:130`** — grants are "a human decision" (F1). **`ADR-0004:134`** — blob caps listed as implemented (F11). **`ADR-0004:94/135`** — revoke documented Owner-only; `main.rs:1213` gates on `can_manage_members` (≥ Admin). ADR-0006 supersedes the last one but ADR-0004 carries no pointer.
4. **`crates/passbook-wasm/src/lib.rs:10-12`** — *"The browser prototype is master-password only; wiring the device Secret Key (2SKD) is a planned follow-up."* 2SKD is fully wired and in use on every seal and open. Safe-direction drift, but it points the reader at `seal_vault`, which contradicts it 140 lines later.

## 3. Refuted candidates worth recording

Recording these so the auditor does not re-tread them. Each was checked in source.

| Candidate | Why refuted |
|---|---|
| **Nonce / CSPRNG reuse** | Clean negative, verified independently. All seven AEAD seal sites draw a fresh 24-byte nonce from `OsRng`; `sealing.rs` and `vault/lib.rs` additionally draw a fresh salt per seal (so the key is re-derived even under nonce collision); `seal_to`/`wrap_to` generate a fresh ephemeral X25519 keypair *per recipient*, so the wrapping key is unique before the nonce matters. `share_to` loops `wrap_to`, giving no cross-recipient reuse. `getrandom` 0.2 `js` feature is genuinely linked in the shipped `.wasm` (verified against `passbook_wasm.js`, not the manifest); the crate fails to compile on wasm32 without it, so there is no silent-weak-entropy build. The only non-CSPRNG call in the tree, `Math.random()` for `member_id`, feeds no key, nonce, or KDF input. **Nothing here.** |
| **No auto-lock / master password retained in Pinia store** | `threat-model-passbook.md:123-129` names ADV-3 *"Device compromise (malware, **stolen unlocked device**, hostile browser extension) … **Explicitly out of scope by design**."* The scenario is the adversary's own title. And the fix would not help: `lock()` deliberately does not clear the Secret Key from `localStorage`, so an attacker at the unlocked tab retains both factors either way. Retention is also structural — `persist()` re-seals from the password on every mutation. Auto-lock is a legitimate product feature; it is not a defect. *(One doc nit: the asset table says A2 lives "transiently in a client call stack." It lives in store state for the tab's lifetime.)* |
| **Clipboard never cleared** | Factually true and broader than reported (card numbers, CVV, TOTP, invite codes, and `ExportDialog.vue:77` puts the entire plaintext export on the clipboard). But: requires a local adversary who already reads the Secret Key, member X25519 secret, and bearer token from plaintext `localStorage`; and the two vivid vectors (Win+V history, macOS Universal Clipboard) capture at *write* time, so a later `writeText('')` cannot retract them — the correct control is a concealed-type clipboard flag, which `navigator.clipboard` does not expose. Worth a `known-limitations` line; not audit-grade. |
| **`SecretKey` does not zeroize** | True (`domain.rs:130-131`, `#[derive(Clone)]` only, against the header's "Secret-bearing fields zeroize on drop"). Zero marginal capability: the same key is deliberately persisted in cleartext `localStorage` and, on the CLI path, written to a plaintext file. Already in `known-limitations.md` §13. Comment fix, not a finding. |
| **Argon2 params not in the sealed envelope / no upgrade path** | The "vaults become permanently un-openable" premise is false. `SealedVault` derives `Serialize/Deserialize` with no `deny_unknown_fields` and is persisted as JSON; `#[serde(default)] params` is a textbook backward-compatible migration, and `seal()` generates a fresh salt every call so a cost bump rolls forward naturally. The candidate also concedes there is no downgrade attack — parameters are compile-time constants, never read off the wire. Separately: `Argon2::default()` is the *current OWASP recommended* configuration, not a deficient one; the self-flagging comment at `crypto/lib.rs:13-14` ("prototype-grade parameters") understates what the code ships. |
| **Non-atomic rotation bricks the vault permanently** | The two-write sequence and absent rollback are real (`sharing.ts:625-627`). "Unrecoverable" is not: the write that *succeeds* is the new keys blob, so `saveFamilyEntries` picks up the new key, and because the failed `putContent` never bumped the version the `If-Match` still matches — any subsequent edit from the still-open tab restores it. `share.ts:357-366` catches and does not call `reloadActive()`, so `active.entries` survives. Availability wart with a working repair path, already in `known-limitations.md` §5. |
| **`revokeMember` re-reads the directory from the untrusted relay** | The stated attack is defeated by the cited line: `sharing.ts:619` filters on exactly the `memberId` just DELETEd, so a replayed pre-DELETE directory has the revoked member stripped. The residual (relay returns the revoked party's key under a *fresh* id) is F1 with extra steps and strictly less payoff. Report F1, not this. |
| **`shuffle` bug as a security finding** | Kept as F12 at low severity. The `length` parameter is never attacker-influenceable: the only in-app path is a slider capped at 40, and no code derives generator length from any remote or untrusted input. Self-inflicted hang, not a vulnerability. |
| **No rollback / fork detection** | Real property, but already stated verbatim in `threat-model-passbook.md:106` and `known-limitations.md:216`, and one escalation claim is wrong — rolling back `/keys` and `/vault` re-exposes only content the revoked member already held, not material written after rotation (that is sealed under `K_vault'`, which the stale blob does not wrap). |
| **Single-threaded loop / blocking Stripe call as *critical*** | The mechanism is real and I confirmed the absent socket timeouts (`ureq` 2.12.1 `timeout_read: None`; no `set_read_timeout` anywhere in `tiny_http` 0.12). But the headline exploit — stall the loop by not reading a 16 MiB response — is defeated by Traefik's `buffering` middleware, which unconditionally buffers the *response* (that is how it can enforce a limit), so the pod's write returns promptly. The Stripe endpoint additionally 503s unless `PROCTOR_STRIPE_SECRET_KEY` **and** `PROCTOR_STRIPE_PRICE_FAMILY` are set; neither appears in any manifest. Genuine hardening (add `.timeout()`, rate-limit the route, cap bodies application-side) — not the outage primitive described, and self-hosters running the binary bare are the real exposure. |

## 4. Where AI review is structurally weak here

Being specific rather than generic, and flagging the shared-blind-spot risk directly: **this codebase was largely written by an AI, and this pre-audit was performed by an AI.** That is a correlated-failure condition, not a hypothetical. Concretely:

**a) The docs are contaminated as evidence, and I could not fully escape them.** `docs/security/known-limitations.md` is unusually candid — it names most of F1–F3 accurately, with file:line. That candor produced a specific pathology across the nine review lenses: roughly a third of all refutations were of the form "already documented, therefore not a finding." That reasoning is wrong for a live service, and it cuts the other way too — several lenses derived their "evidence" from doc prose rather than from code, and inherited stale claims (§1 `was_contributory`) as fact. A human auditor reading only source would not have this bias in either direction. **Treat every doc citation in this report as a pointer, not as evidence.**

**b) Composition across trust boundaries is where I am weakest, and it is where the real bugs are.** Every individual primitive here is correctly implemented — the Argon2id call, the HKDF labels, the AEAD constructions, the low-order-point rejection, the nonce discipline. The breaks are all in *composition*: Rust-side `grant_access` is a correct primitive that the TypeScript caller invokes in a loop over attacker-controlled input; `safety_number` is a correct digest computed at the wrong point in the sequence; `apply_redeem` is correct policy with the wrong uniqueness key. I found these by tracing call sites, which is mechanical. What I cannot do reliably is reason about the *protocol as a whole* — whether the state machine of `{create, join, grant, rotate, revoke, re-join}` has reachable states nobody enumerated. **F6 (permanently ownerless group) is the shape of thing that falls out of that analysis, and I found it only because a lens happened to trace one path.** Assume there are more.

**c) I verify what is present; I am poor at noticing what is absent.** No sender authentication on wraps (F2) was found only because a lens explicitly asked "what does the safety number cover?" There is no equivalent prompt for "what should this protocol have that it doesn't." A cryptographer comparing this construction against the Signal/MLS/Keybase design space would spot the absences — no key epochs, no ratchet, no transparency log, no member-signed directory — as a class, in minutes.

**d) I cannot assess parameter adequacy against real hardware.** I can report `m=19456 KiB, t=2, p=1` and match it to the OWASP table. I cannot tell you what that costs on 2026 GPUs, or what a browser-main-thread derivation budget should be for a password manager rather than a web app. `review-scope.md` Q7 asks exactly this. Pay for the answer.

**e) I did not exercise anything.** Every claim here is static reading plus arithmetic. I did not stand up the relay and prove F1 end-to-end, did not fuzz `SharedVault`/`ContentBlob`/`SealedVault` deserialization, did not check any construction against test vectors, and did not measure timing on `unwrap_for`'s member lookup or the AEAD paths. `known-limitations.md` §12 correctly notes there is no fuzzing and no test vectors — that gap is entirely unaddressed by this document.

**f) The one place I'd most expect a shared blind spot.** The `member_id`-as-identity design (F7) is the kind of choice an AI makes fluently and reviews approvingly: it is internally consistent, well-commented, and wrong at the seam where a client-chosen string becomes a cryptographic identity. The Postgres PK that accidentally saves the managed instance is the tell — one adapter enforces an invariant the shared "pure so both adapters share the exact policy" function does not, and the comment asserting that shared-policy property is itself false. **Ask the auditor to look specifically for other invariants that are enforced incidentally by a storage layer rather than deliberately by the policy layer.**

## 5. What to hand the auditor to reduce cost

**Do these before the engagement starts — they are cheap and they directly buy hours:**

1. **Resync `known-limitations.md` §1, §2a, §10a and `review-scope.md` Q2/Q3 with the code.** `checked_secret` and `SEALED_BOX_INFO` exist and are tested. As written, the package's stated #1 priority is a PoC against a closed hole. This single edit is worth more than anything else in this section.
2. **Fix the four doc/code contradictions in F13.** An auditor who catches you being wrong about your own code — in either direction — will (correctly) stop trusting every other document you hand them, and re-derive everything from source at your expense.
3. **Fix F12 (one line) and F9 (wordlist swap + structure-aware scorer).** Both are unambiguous, neither needs a cryptographer, and F9 is a user-facing correctness failure you do not want discovered by an auditor billing by the hour.
4. **State the deployment reality up front.** ADR-0004 makes external review a hard gate before GA; the managed instance is live at `proctor.klarlabs.de`. Say so in the first paragraph of the engagement brief, with the current user count. The auditor's severity model depends entirely on it, and discovering it themselves reads badly.

**Hand them:**

5. **This report, with F1–F3 marked as "we believe these are broken; confirm the fix, don't re-derive the bug."** The expensive question is not *whether* the relay can steal `K_vault` — it can. It is whether TOFU pinning + blocking safety-number confirmation is the right fix, or whether member-signed directory entries are required. That is a design question worth paying for.
6. **A written protocol description for the group state machine** — the reachable states of `{create, join, grant, rotate, revoke, re-join, recovery-contact}` and the invariant each transition is supposed to preserve. This does not exist today; ADR-0004 describes the happy path. F4, F6, and F7 are all invariant violations in that machine, and a single page of state/invariant documentation would let an auditor check the whole space in an hour instead of reconstructing it from handlers.
7. **The exact deployment topology as a fact, not a manifest.** Postgres backend (which silently mitigates F7), Traefik with the `buffering` middleware and the `metrics-block` `ipAllowList`, NetworkPolicy source restriction, replica count and HPA range. Three of the nine review lenses reached wrong conclusions about exploitability purely from reading base manifests without the `overlays/klarlabs` layer. Write the effective topology down once.
8. **A regression-test index mapping each `known-limitations` entry to the test that closes it**, so "is this still open?" is a `cargo test` away rather than a code read. `low_order_public_keys_are_rejected` and `recovery_and_vault_wrapping_use_separate_derivations` already exist and already close two entries the docs still list as open — that index would have caught F13.1 automatically.
9. **Explicit scope guidance on what you have already accepted vs. what is merely undisclosed.** Device compromise (ADV-3) is genuinely out of scope and the docs are consistent about it — say so loudly, because three refuted candidates in this report burned effort re-litigating it. But be precise that "documented" ≠ "accepted": F1–F3 are documented *and* named as pre-GA blockers, and you want a verdict on the fix, not a restatement of the problem.
